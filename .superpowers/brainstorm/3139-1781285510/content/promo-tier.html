Promotion tier system — how far do we go?

The spec defines 5 tiers. Your DB only has a binary isPromoted flag. Pick the approach.

A

Keep binary — adapt the spec

Two visual states only: plain and promoted. Promoted cards get an amber frame + "Boosted" badge. No schema change needed — ships today.

Pros

  • Zero migration
  • Matches existing Paystack packages
  • Simpler UI to maintain

Cons

  • Less visual differentiation
  • Can't upsell diamond/enterprise later without changes
B

Full 5-tier system (spec as written)

Add promotionTier enum to Prisma schema: NONE | BOOST | TOP | DIAMOND | ENTERPRISE. Each tier gets distinct card chrome — frame colour, badge, rail label for diamond/enterprise.

Pros

  • Matches the spec exactly
  • Visual revenue hierarchy is clear
  • Future-proof for upsell flows

Cons

  • DB migration + Paystack package update needed
  • More complex promo-purchase UI
  • 1–2 extra days of work
C

3-tier shortcut (none / boost / featured)

Add isFeatured boolean (already in schema for stores). Map: plain = none, isPromoted = boost badge, isFeatured = diamond-style gold frame. No new enum needed.

Pros

  • Richer than binary, cheaper than 5-tier
  • Reuses existing isFeatured field
  • Minimal migration

Cons

  • isFeatured was meant for stores, not listings
  • Slightly awkward semantics

Recommendation: A to ship fast, B if you want the full revenue model now.
