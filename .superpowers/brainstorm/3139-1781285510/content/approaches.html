Implementation approach — how do we sequence this?

Full scope confirmed: 5-tier promo, attribute grid, market price, seller ratings. Three ways to get there.

A

✦ Recommended — Schema-first, then UI in two waves

Wave 1 (schema + feed): All migrations in one PR (promotionTier, attributes, market price columns, SellerReview table). Redesign ListingCard with tier chrome + grid/list toggle. Ship.

Wave 2 (detail page): Full detail page redesign — attribute grid, market-price band, lazy contact reveal, seller rating card, quick-reply chat openers.

Pros

  • Schema done once, both waves build on it
  • Wave 1 ships visible wins fast (card chrome)
  • Wave 2 is self-contained detail work

Cons

  • Two PRs, not one
B

Big bang — everything in one implementation plan

One plan covers schema migrations, backend updates (promo logic, market-price job, review API), ListingFeed redesign, and full ListingDetail redesign. One large PR.

Pros

  • Single coherent diff
  • No intermediate "partial" state

Cons

  • Very large context — higher risk of drift
  • Nothing ships until everything is done
  • Harder to review
C

UI-first — fake the new features, wire real data later

Build the full UI with mock/hardcoded data first (promotionTier mapped from isPromoted, no real attribute grid). Get visual sign-off. Then add schema + real data in a second pass.

Pros

  • See the full UI very fast
  • Easy to iterate on design before committing to schema

Cons

  • Throwaway work when wiring real data
  • Risk: UI looks great but schema turns out not to fit
